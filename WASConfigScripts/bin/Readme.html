The zip file contains examples to change, export, and import existing configuration data.  The following example tasks are provided:

      - change host name

- change cell name

- change node name

- change server name

      - export/import the whole repository config directory 

      - export/import individual application server

- export/import enterprise application 

- export/import JDBC Provider with associate Data Sources

 

Supported version:

      WebSphere version 6 or above

      In case of mixed cell environment, WebSphere Deployment Manager version must be version 6x and

      - Copy all the files in …/ConfigScripts/bin directory to your <was root>/bin directory.

 

Pre-run do list:

      - Since the scripts are run in local mode, it is very important that server1 process (for Application Server environment) or dmgr    process (for Network Deployment environment) is stopped. Otherwise you may end up having inconsistent or corrupted configuration.

 

- For export/import on Network Deployment environment, install and federate all nodes before the source configuration is laid over the target cell.

       

- For export/import, it is user responsibility to move the export directory on the source machine to import directory on the target machine before running import task on target machine.

3. How to change node name

 

!!!!! IMPORTANT WARNING !!!!!

 

An important consideration related to modifying the node name is that WebSphere Embedded Messaging uses the node name to identify the message broker and queue manager processes. In order to change the identity of these processes, their definition must be destroyed and recreated.  This would have the effect of losing all the messages currently on Queues or Topics.  Therefore, all messages should be drained from the effected Queues or Topic before changing the node name.  Once you have done that, you will find you still can't connect to the JMS Server from applications unless you change all your Queues and Topic Connection Factories definitions since the node name is one of their attributes.  For this reason, it is particularly discouraged to change the node name after install when WebSphere Embedded Messaging is in use.

 

If you have installed WebSphere Embedded Messaging and still insist to change node name, in addition to all renaming tasks you need to:               

- Drain down the messages on any Queues/Topics (this must be done by running application code that processes all messages).

- Run the deletemq utility to remove the message broker and queue managers using the old nodename.

      - Run createmq to recreate definitions using the new nodename.

- Change the node name on all the relevant Queue and Topic Connection Factories to the new node name.

1. How to export/import the whole config repository

 

1.1 Additional assumptions used in export/import the whole config repository:

 

  - Identical source and target node operating systems

 

If the operating systems for any of the nodes in the target cell are different from the node in the source cell, then additional modification must be made to the configuration files.  Specifically, the value of the $WAS_INSTALL_ROOT, $JAVA_HOME, $MQ_INSTALL_ROOT and any other user defined variables must be altered to match those applicable to the operating system on the target node.  These variables are defined in the variables.xml file in the directory

containing the node configuration data eg. <WAS_HOME>/config/cell/myCell/node/myNode.  If those variables are defined in a different scope, then those variables need to be changed too.  It is user responsibility to update those values on the affected target node.

 

  - For network deployment environment, there should be identical number of nodeagents in source cell and target cell.

 

  - Identical fixpack levels for both source and target nodes

 

Repository migration may work between nodes with different fixpack levels, but user should consult the release notes

for both levels of the product to be sure.  Certain fixpack levels are known to include changes that require  

different configuration files or configuration file contents (for instance 6.0.0 and 6.0.1).  Repository migration may

not be possible between cells (or portions of cells) at different fixpack levels.

 

- If possible, keep the target cell the same topology as the source cell.  It is recommended that the names of the nodes in the target cell the same from the source cell.  There is no problem if nodes in source cell are the same name as nodes in target cell.  If it happened to be different names, then user need to run changeNodeName task after running the import task to match with the topology in the target machine.                                                         

 

- After running export task on the source machine, user is responsible to transfer the entire export directory from the

source machine to the target machine where import is to be performed. Make sure that the export directory structures are maintained and all the files and subdirectories are transferred.

 

  - The export/import example scripts are run using the ws_ant command.

          ws_ant[.sh] [–profileName xxx] –buildfile xxx [-logfile xxx] –DnodeRelease=5|6

                      –DmanagerHostName=xxx –DmanagerSOAPPort=xxx importPart2

 

          Example:

 

          ws_ant [–profileName V6nodeProfile] -buildfile exportImport.xml –logfile import.log

                 -DnodeRelease=5 -DmanagerHostName=Goodhost.ibm.com –DmanagerSOAPPort=8879 importPart2   

 

      2.2.2.4 Post-execution checklist for Application Server environment or Network Deployment environment

              For Application Server environment:

                  Make sure the SOAP port number of the imported unmanaged server is the same as the one in                             wsadmin.properties file.                 

                  Otherwise user need to update the com.ibm.ws.scritping.port property to the SOAP imported managed server              port in wsadmin.properties file.

 

              For Network Deployment environment if the imported server is dmgr server:

                  Make sure the SOAP port number of the imported dmgr server is the same as the one in wsadmin.properties

                  file. 

                  Otherwise user need to update the SOAP port number in wsadmin.properties file on all the node machines.

 

 

3. How to export/import an Enterprise Application

 

3.1 Additional assumptions used in importing an enterprise application:

 

- Application is deployed to the same node(s) and server(s) as it was in source machine.  It is user's responsibility to make sure that all the node(s) and server(s) that will have this application imported exist in the target machine.

 

  - Export/import enterprise application task do not include the resources that may be used by the exported application.

It is the user's responsibility to make sure that the resources used in the application are available in the target

machine.

 

3.2 Export/Import an Enterprise Application on WebSphere Application Server environment or Network Deployment environment

                 

3.2.1 Export an enterprise application on source machine

 

      3.2.1.1 Pre-execution checklist

 

        Stop all running servers including deployment manager and nodeagent processes on source machine before performing

        the export task.

 

3.2.1.2 ws_ant task

 

  Syntax:

 

        ws_ant –profileName xxx -buildfile xxx -DexportDir=xxx -DappName=xxx exportApplication

 

        Example:

 

        ws_ant –profileName dmgrProfile -buildfile exportImport.xml -DexportDir=/myuser/applicationExport

               -DappName=myapp exportApplication

  

      3.2.1.3 Post-execution checklist

        Move the export directory from source machine to target machine.

 

    3.2.2 Import an enterprise application on target machine

There are two ws_ant tasks that you need to run to import an enterprise application in network deployment

environment.  The first task needs to be run from deployment manager machine.  Additional task needs to be run on

node machines that install the imported application. This task includes synchronization of the configuration from

      the deployment manager to the nodes. 

 

      3.2.2.1 Pre-execution checklist

 

        - Back up your configuration data <WAS_HOME>/profiles directory.

        - Stop all running server(s) (including deployment manager and nodeagent processes in Network Deployment

        environment) on target machine before performing the import task.

 

3.2.2.2 ws_ant task

        This task is to be run in both Application Server and Network Deployment environment.  For Network

        Deployment environment, the task needs to be run from deployment manager machine.

 

        Syntax:

 

        ws_ant –profileName xxx -buildfile xxx -DimportDir=xxx -DappName=xxx [-Dparam=xxx] importApplication

 

- If the application installed on source machine have different operating system platform than the target machine, then you need to provide the param property.  This property should have a value of applicationname+BinariesURL=<directory path to install the application on target machine>.

 

For example:

  myapp is installed on C:/temp of source Windows machine.  This application is going to be imported to a

  target machine that has UNIX machine.

 

  You need to pass in the param property to the task as below:

 

          Example:

 

          ws_ant –profileName dmgrProfile -builffile exportImport.xml -DimportDir=/myuser/applicationExport

                 -DappName=myapp –Dparam=myapp+BinariesURL=/usr/temp importApplication

  

3.2.2.3 Post-execution checklist for Network Deployment environment

              Start dmgr process in target cell by running startManager[.sh]

 

        3.2.2.3.1 Run task from target node machine(s) that install the imported enterprise application

                  -DnodeRelease option is required to specify the version of the node you are running it from.  If the node version is                    WebSphere version 5x then you need to use –DnodeRelease=5.  If the node version is WebSphere version 6x then you need to                         use –DnodeRelease=6.

                  If your node is WebSphere Version 6 then –profileName option is required. 

                  If your node is WebSphere Version 5 then you can’t use –profileName option.

 

          Syntax:

 

          ws_ant[.sh] [–profileName xxx] –buildfile xxx [-logfile xxx] –DnodeRelease=5|6

                      –DmanagerHostName=xxx –DmanagerSOAPPort=xxx importPart2

 

          Example:

 

          ws_ant [–profileName V6nodeProfile] -buildfile exportImport.xml –DnodeRelease=6

                 -DmanagerHostName=Goodhost.ibm.com –DmanagerSOAPPort=8879 importPart2    

 

 

4. How to export/import a JDBCProvider and associated DataSources

 

4.1 Additional assumptions used in exporting JDBC Provider:

 

  - Mapping modules in exported Data Sources are not exported.

 

  - Data Sources associated with the exported JDBC Provider are exported.

 

- Relational resource adapter associated with the Data Sources is not exported with the assumption that the built-in

default resource adapter is available in the target machine and will be used by the imported Data Sources.

 

- If Data Source contains the identifier for the auth data entry, the associated auth data entry which contains the

user id and password for use by the Java 2 Connector for the Data Source are not exported.

 

- No configuration files are exported. Before see "Additional assumptions/considerations in importing JDBC Provider" on

how to modify information associated with the imported JDBC Provider and associated DataSources.

 

4.2 Additional assumptions used in importing JDBC Provider:

 

- Since the export of JDBC Provider does not create any configuration file, you can change the following types of data

during import:

 

  + JDBC Provider classpath

  + JDBC Provider native path

+ DataSource auth data alias (This single alias will be used to update all the DataSources which have auth alias set.

This will not update DataSource that does not set auth data alias.)

 

You can modify the other information once the import is done by using $AdminConfig modify command on the imported

object. 

 

 - If an imported DataSource contains the identifier to point to an auth data entry, it is the user's responsibility to

 setup such auth data entry containing the same identifier.
